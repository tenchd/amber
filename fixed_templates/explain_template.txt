3. Scope
This timestamp has three components: this document, the Merkle tree, and the blockchain message. Taken together, they comprise a secure timestamp of the ${corpus_name} corpus.
The function of this secure timestamp is:
- To provide cryptographic-strength, censorship- and tamper-resistant proof that each text in ${corpus_name} existed on ${date}. This can be used to re-establish trust in provenance (see Section 4).

This secure timestamp is NOT intended to:
- independently prove that any text in the timestamped corpus is authentic (not a forgery, correctly translated, complete, etc.).

This timestamp was made assuming the following conditions continue to hold in the future:
- Finding a SHA256 hash collision remains computationally infeasible.
- Genuine, unaltered versions of the texts that comprise the corpus survive (and ideally are stored alongside this timestamp).
- This document, and the accompanying Merkle tree, survive.
- The message written on the Bitcoin blockchain containing the Merkle root hash and the hash of this document remains readable and provably unaltered (i.e., the blockchain is not edited in some kind of 51% attack).

4. Trust in Provenance
Technically speaking, this secure timestamp establishes the fact that each text in the corpus existed in ${date}, but nothing else. For example, if one of the texts in the corpus was a text file of the novel Don Quixote, this timestamp would prove that the text file was not created after ${date}. However, it would not conclusively prove that it was truly a novel published in the early seventeenth century by Miguel de Cervantes, or that it was a faithful and accurate translation into English from the original Spanish. It also would not rule out the possibility that a malicious actor modified the text shortly before it was timestamped.
Despite these limitations, proof of existence by ${date} alone is likely to be useful for many authentication purposes. There are two primary reasons for this which we summarize now.
1) Coherently altering a large corpus of historical documents is currently expensive, so proving that a corpus existed today means that it could only have been tampered with at great cost.
For example, imagine that the government of Country A wishes to rewrite the historical record of its centuries-long series of wars with Country B. While Country A may not be able to destroy all existing genuine digital copies of historical texts concerning these wars, they could widely circulate altered versions of these digital texts to downplay A's misdeeds and exaggerate those of B. If successful, it might be difficult for future historians to distinguish between two conflicting stories, each supported by a large corpus of internally-consistent texts. However, at the time that the Amber team is writing this template, in late 2026, coherently altering a large corpus of texts in this manner is expensive. For instance, A may wish to claim (falsely) that ; it requires many hours of expert human labor to accomplish, and existing LLM technologies are unlikely to reduce the cost significantly due to PUT REASON HERE.
2) If there is a technological arms race between AI-powered forgery techniques and methods to detect forgeries, proving that a corpus existed before the latest forgery techniques greatly simplifies the task of establishing authenticity.
Make the point that if the detection methods can detect forgeries using techniques 5 years old or older, timestamped data is very valuable because it establishes the data existed only when detectable forgery methods were available; if no such methods are detected one can conclude the data was not forged.

6. Merkle Tree Construction and Serialization
The Merkle tree is constructed in the same way a Bitcoin miner computes the Merkle root hash of a block of transactions. That is to say, define an arbitrary ordering over the files in the corpus. First, compute the double SHA256 hash of each file. These comprise the leaves of the Merkle tree. Then for each adjacent pair of leaves in the ordering, double hash the concatenation of their hashes to produce the hash of their parent node. The process is repeated at each level until the root is reached. If there is an odd number of nodes at some level, the last node's hash is concatenated with itself and then double SHA256 hashed to get the parent hash.

The Merkle tree is written to file according to the following specification, which is designed to be human-readable, relatively compact, simple, self-explanatory, and also to be machine-readable by Amber version ${version}. First there are seven header lines that read:
"# Merkle tree. Created on <date> from Project Gutenberg corpus of plain text files."
"# Number of leaves: <num_leaves>"
"# Root hash written to Bitcoin blockchain along with identifier <identifier>"
"# Block height: <block_height>"
"# Tx hash: <tx_hash>"
"# Explain.txt hash: <explain_hash>"
"# Each line below is the hash (in base64) of a merkle node. Tree is binary. Each parent hash is the double SHA256 hash of the concatenation of its two child hashes. If the parent has only one child, its hash is the double SHA256 hash of the child hash concatenated with itself. Final line of file is root hash."
In these header lines,
- num_leaves indicates the number of leaves in the Merkle tree.
- identifier indicates the 8-character ASCII string that begins the blockchain message.
- block height refers to the actual block in the Bitcoin blockchain where the message is written. This can be used to location the blockchain message that timestamps the corpus.
- tx hash refers to the unique hash of the transaction that wrote the message to the blockchain; this can be used to identify the transaction. 
- Finally, explain hash refers to the double SHA256 hash of this document that you are reading now. 

This is followed by the list of hashes in the Merkle tree. First the leaf hashes are listed in order. Next the hashes of the nodes in the next level are listed in increasing order of left child. This is repeated for every level until the root, which is the last hash written to the file.

Amber version ${version} reads this file format automatically and can build and verify the Merkle tree, construct the expected blockchain message, and confirm that this message exists in the specified block and transaction on the blockchain, confirming the timestamp.

It's not possible to be sure a priori which block a transaction will end up in due to the way Bitcoin mining works, but the Merkle tree file (whose header lines were updated after the transaction was successfully written to the chain) should contain the exact block, which should be shortly after ${locktime}.

8. Verification
Let's say you have this document, a file containing the Merkle tree, and some text files. You would like to make sure that the text files were part of the ${corpus_name} corpus on ${date}.
First, you must verify that this document and the Merkle tree are valid. To do this, first verify that there is a transaction on the Bitcoin blockchain in block ${locktime} or shortly thereafter containing the message described above. It should exactly match the block height listed in the Merkle tree file header. The Merkle tree root hash in the blockchain message should exactly match the root hash written in this file, and the SHA256 double hash of this document should match exactly as well. Further, the Merkle tree should be valid (meaning the hash relationships between nodes are correct) and the root hash should match the one written in this file. The Merkle tree should have ${num_merkle_leaves} leaves.
If the above conditions hold, you can be sure the Merkle tree is valid, and can use it to verify corpus texts safely.
To verify that a text file was part of the PG corpus in 2026, simply double SHA256 hash the file. If there is a leaf node in the Merkle tree whose hash matches exactly, you have verified that the text file is part of the corpus. Otherwise, the text file cannot be verified, and you can conclude that it does not exactly match any of the original files included in the Merkle tree.

9. Preservation and Dissemination
This doc and the original corpus files should be kept together and preserved exactly, because any slight alteration of them changes their hash signature and therefore makes it impossible to verify their timestamp using the blockchain message and Merkle tree. The hash of the Merkle tree file is not included in the blockchain message because some of the metadata stored in its headers can only be written after the blockchain message is created and written to the Bitcoin chain. This means that you can technically make certain small alterations to this file and still verify successfully, provided you do not tamper with the leaf hashes or the metadata stored in the header lines. However, we strongly discourage you from altering this file unless you have an extremely good reason to do so.
